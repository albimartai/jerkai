## 0. Scope & Sequencing Note (read first)

v1.1 is a revision of the shipped v1 dashboard. v1 validated the read surface; in daily use, the stack below the body-fat strip reads as noise. Root cause: the north-star strip applies the raw-dots-plus-trend-lines treatment, while every other strip plots raw daily values as its primary mark. v1.1 applies one rendering rule everywhere, promotes weight to a strip, and demotes Recovery Score from strip to summary readout.

Decision-log prerequisites (recorded before build; entries live in the project decision log, kept in the Career vault, not this repo):

* DL-2026-07-18-a — Recovery Score demoted from main-stack strip to guardrail readout + Whoop-detail strip. Supersedes the v1 strip stack (DL-2026-07-16-d layout consequence).
* DL-2026-07-18-b — Weight strip added to the main stack (Fitdays, already ingested). Rationale: less noisy than BF%, and the series the future TDEE Engine slice annotates.

What this slice is NOT: no Calories-vs-target strip (still gated on Log Meal), no header CTA changes (AC-D14/D15 unchanged), no stall-badge changes (AC-D4–D7 unchanged), no new ingest, no TDEE math.

v1.1 main stack (top → bottom): Body fat % (north star, tallest) · Weight (new · Fitdays) · Day Strain trend (driver · Whoop) · Lean body mass (guardrail · Fitdays) · guardrail readout row · collapsible Whoop detail (HRV, RHR, sleep, Recovery Score).

## 2. Problem statement

The v1 dashboard answers "what changed when the body-fat trend changed?" — but only the body-fat strip is legible at a glance. Day Strain (spiky 0–21 dailies), Recovery Score (the noisiest series on the page), and lean body mass (BIA noise on a wide Y domain) all render raw daily values as primary marks, so the vertical scan the dashboard exists for degrades into visual noise below the first strip. The user reports checking only the top strip; the driver and guardrails are not earning their screen space.

## 3. Objective

One scan, four charts, one readout row. Every plotted series follows the north-star treatment (faint raw, dominant trend line); guardrails that only need a yes/no ("is recovery holding? is lean mass holding?") become summary readouts instead of charts. Success = the driver strip becomes usable for stall investigation, and nothing on the default view renders raw daily values as its primary mark.

## 4. Functional Requirements — user-centered acceptance criteria

Written as testable Given/When/Then. "The user" = Albert (single user).

### 4.1 Global rendering rule

* AC-N1 — Given any strip that plots a daily-sampled series, When it renders, Then raw daily values appear as low-emphasis dots and a 7-day rolling line is the visually dominant element. No strip renders a raw daily line as its primary mark.
* AC-N2 — Given fewer days of history than a rolling window, Then averages compute over available days only; no errors, no artificial gaps beyond genuine source-side gaps (extends AC-D3 to all strips).

### 4.2 Strip stack

* AC-N3 (body fat — unchanged) — Raw dots + 7d/30d lines + legend; hero shows today's raw plus both trends; raw never hidden (AC-D1/D2 remain in force verbatim).
* AC-N4 (weight — new) — Given Fitdays weight data exists, When the dashboard renders, Then a weight strip appears directly below body fat with raw dots + 7d/30d lines, participating in the shared axis, window toggle, and hover-scrub like any other strip.
* AC-N5 (strain — amended) — Given Whoop Day Strain data, When the strip renders, Then daily strains are faint dots on a fixed 0–21 domain and a 7-day rolling strain line is the primary mark. Hover readout shows both, e.g. "Jul 12 · 14.2 · 7d 12.8". Source unchanged: `source='whoop'`, `metric='day_strain'` (amends AC-D11 rendering; source clause unchanged).
* AC-N6 (recovery — demoted) — Given the dashboard renders, Then no Recovery Score strip appears in the main stack. Recovery Score appears as (a) a guardrail readout per AC-N9 and (b) a full strip inside the collapsible Whoop detail, rendered per AC-N1 with zone bands.
* AC-N7 (lean mass — amended) — Given Fitdays lean-mass data, When its strip renders, Then it uses faint raw dots + 7d/30d lines, displayed in lb, with the Y domain fitted to the observed range ± a fixed margin (not zero-based), so genuine drift is visually detectable.

### 4.3 Guardrail readout row

* AC-N8 (lean-mass readout) — Given ≥30 days of lean-mass data, Then the readout row shows the 30-day lean-mass change in lb with a directional state: within ±0.5 lb = "holding" (neutral/good color), below −0.5 lb = warning color (confirmed 2026-07-18; ≈ the ±0.2 kg band). Thresholds live in config, not hardcoded. With <30 days, the readout shows the available-window change labeled with its actual span.
* AC-N9 (recovery readout) — Given Whoop Recovery data, Then the readout row shows the 7-day average Recovery Score plus the count of red-zone days in the last 7, e.g. "Recovery 7d 68% · 2 red days".
* AC-N10 (readout row placement & passivity) — The readout row renders in exactly one location (directly below the strip stack), each readout is a summary statistic and never a chart, readouts do not respond to hover-scrub, and no readout asserts a cause (extends AC-D7).
* Superseded: AC-D12 (Recovery + lean-mass guardrail strips) is superseded by AC-N6–N10.

### 4.4 Scrub & detail

* AC-N11 — Given the user hovers any strip, Then the crosshair spans all main-stack strips — including the new weight strip — and, if expanded, all Whoop-detail strips (extends AC-D8/D9; AC-D10 return-to-summary behavior unchanged).
* AC-N12 — Given the Whoop detail is expanded, Then HRV, RHR, sleep, and Recovery Score strips all follow AC-N1 (faint raw dots + 7-day line as primary mark).

### 4.5 Noise-reduction outcome (the point of the slice)

* AC-N13 — Given the dashboard loads with the Whoop detail collapsed, Then exactly four charts are visible (body fat, weight, strain, lean mass) plus one readout row; no other chart renders. (Fallback if lean mass proves unreadable even under AC-N7: demote to readout-only — but that is a decision-log call made with the user, not a silent build choice.)
* AC-N14 (outlier absorption — regression check) — Given synthetic fixture data with a known injected single-day outlier in weight or lean mass, When strips render, Then the 7d/30d lines visibly absorb the outlier while the raw dot remains plotted — provable with a snapshot/fixture test.

### 4.6 Unchanged and in force from v1

AC-D1–D10 (except as amended above), AC-D13 (gaps not zeros), AC-D14/D15 (header), AC-D16 (window toggle), AC-D17 (responsive). Tests covering these must stay green — this slice must not regress them.

## 5. Non-Functional Requirements (FDE)

Aligned to the stack in [README.md](../../README.md) (Next.js App Router + TS, Neon Postgres, Vercel, Auth.js magic-link, Whoop OAuth, Health Auto Export ingest). All v1 NFRs (NFR-1–13) remain in force; the following are additive.

* NFR-14 — One strip component. Body fat, weight, strain, and lean mass render via a single parameterized strip component (series config: domain, windows, emphasis). If the diff introduces a second strip implementation, the build is wrong.
* NFR-15 — Read-only slice. No schema changes, no new migrations, no ingest changes. Weight and Recovery data are already in `biometric_readings`; v1.1 is a read-path and rendering change only.
* NFR-16 — Config over constants. Guardrail thresholds (±0.5 lb lean-mass band, recovery red-zone boundary <34%) and Y-domain margins live in one typed config module so the user can tune them without touching component code. Display units are imperial (lb) for weight and lean mass regardless of stored source units (README convention: units stored as-sent; convert at read/render time).
* NFR-17 — Derived series computed once. Rolling lines and readout statistics are computed once per window load (reusing the existing rolling-average lib functions), not per hover event; scrub stays within one animation frame (reaffirms NFR-6).
* NFR-18 — Render budget holds. The added weight strip and readout row must not push warm-load interactivity past the existing ~1.5s budget (NFR-5) at either window size.
* NFR-19 — Fixture-based visual regression. AC-N13 (chart count) and AC-N14 (outlier absorption) are asserted against synthetic fixtures in CI, not manual review alone.
* NFR-20 — Test continuity. Extend the existing Vitest units and disposable-Neon-branch integration patterns; reference AC ids in test names. v1 tests are refactored where ACs are superseded (AC-D12) — deleted assertions must map to a superseding AC, never silently dropped.

## 6. Definition of Done

### 6.1 Dashboard v1.1

* Main stack renders exactly: body fat · weight · Day Strain trend · lean body mass · guardrail readout row; Whoop detail collapsible holds HRV, RHR, sleep, Recovery Score (AC-N13, AC-N6).
* Every strip follows the raw-faint / trend-dominant rule, including expanded Whoop detail (AC-N1, AC-N12).
* Weight strip live on the shared axis with scrub and window toggle (AC-N4, AC-N11).
* Strain strip shows 7-day rolling line as primary mark on the fixed 0–21 domain; hover shows daily + 7d (AC-N5).
* Lean-mass strip Y domain fitted to observed range, not zero-based (AC-N7).
* Guardrail readout row implements both readouts with config-driven thresholds (±0.5 lb lean-mass band; recovery red <34%); no hover response; passive language (AC-N8–N10, NFR-16).
* Weight and lean mass display in lb everywhere on the dashboard; stored source units untouched (NFR-16).
* Outlier-absorption fixture test and four-chart-count assertion in CI (AC-N14, NFR-19).
* Single strip component confirmed in PR description (NFR-14); no schema/migration diff (NFR-15).
* All surviving v1 AC tests green; superseded tests refactored with mapping noted in the PR (NFR-20).
* Spot-check: weight, strain 7d, recovery readout, and lean-mass readout values match source apps (Fitdays, Whoop) on ≥3 dates.
* DL-2026-07-18-a and DL-2026-07-18-b recorded in the decision log with final ids before merge — done.
