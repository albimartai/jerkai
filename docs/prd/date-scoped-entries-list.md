## 0. Scope & Sequencing Note (read first)

Edit & Delete Meal shipped the correction path: any logged entry can be edited (including its date) or deleted, and every downstream surface recomputes. AC-M19 works — editing an entry's date to an earlier day correctly detaches it from the old day and attaches it to the new one. But the `/log-meal` entries list computes "today" from the device-local date on mount (`MealEntriesList` calls `todayLocal()` in a mount effect and never changes it) and queries `manual_macro_entries` for that single day. So the instant AC-M19 moves an entry off today, the list has nothing to show it — and there is no in-UI path back to a past day's entries to edit or delete them again. OQ-1 in the predecessor PRD resolved the list as "directly below the form" but explicitly did **not** scope navigation to arbitrary days; today-only was a deliberate, acknowledged scope cut. In practice it means a date-edit is a one-way trip through the UI: re-correcting a moved entry requires direct DB access. That is the exact gap this slice closes.

This is a **pure client-state slice.** No new table, no migration, no new server action, no new surface concept, and no new metric. The one server function the list already calls — `listMealEntriesForDate(entryDate)` in `app/log-meal/actions.ts` — already accepts an arbitrary date and is fully general; the list simply never feeds it anything but today. The whole fix is: lift the page's date into a single owner and let both the form and the list read it, so the page has **one date context** rather than two independently-initialized-to-today states that silently diverge.

The design decision that shapes everything (from Edit & Delete Meal, unchanged): the `/log-meal` page is where you both enter and review entries, and OQ-1 keeps the list directly below the form. This slice honors that — it does not add a competing date control or a second surface; it makes the existing list obey the date the form is already on.

Why not the alternatives (recorded so the build agent does not re-litigate scope): a decoupled list-only date browser (a second date input driving only the list) was rejected because two date fields on one screen invite the "form is on Jul 8 but list shows Jul 21" confusion this slice exists to remove — one date context is the point. A day-detail view off `/daily` is the durable long-term home for browsing/correcting arbitrary days, but it is a genuinely new surface with its own navigation and layout, larger than this gap needs; it is filed as a fast-follow, not built here.

Decision-log (Albert logged in PM/vault, outside the jerkai repo — confirmed 2026-07-21, see [[JerkAI - Decision Log]] DL-2026-07-21-a):

* DL-2026-07-21-a1 (one date context per page, not two) — `/log-meal` has a single source of truth for "which day are we on": the form's `entryDate`. The entries list reads that same value rather than owning its own today-pinned copy. Rationale: the list and form date defaulting independently to today is what created the divergence; unifying them makes the reachability of a moved entry automatic and removes a class of stale-view confusion. The panel (`LogMealPanel`) becomes the owner; form and list are both readers.
* DL-2026-07-21-a2 (after a save, the page's date snaps to the saved entry's date) — When a save (new entry or edit) completes, the page's date context is set to the saved `state.entryDate`. So editing an entry onto Jul 8 leaves the page — and therefore the list — on Jul 8, where the just-moved entry is visible for further edit/delete. Rationale: this is the direct closure of the gap; the correction lands and the corrected row is immediately re-reachable. This does not re-introduce a "delete-and-re-add" pattern (that is forbidden by DL-2026-07-20-b2 and untouched here) — it only moves the *view*, not the data.
* DL-2026-07-21-a3 (no prev/next stepping or calendar in this slice) — Changing the page's date is done through the existing native date input already present in the Log Meal form; no arrow stepper, no calendar popover, no "jump to today" button is built now. Rationale: the native `<input type="date">` already covers arbitrary-day selection at zero new-component cost and keeps the slice thin; ergonomic stepping is a candidate fast-follow if daily use shows friction.

What this slice is NOT: no day-detail view off `/daily` (fast-follow), no second/decoupled date control on the list, no prev/next-day arrows or calendar UI, no "jump to today" affordance, no favorites/recents (model 2d), no changes to targets, the badge, the ledger, any strip, any server action, or the `manual_macro_entries` schema. No migration.

## 2. Problem statement

The correction path shipped, but the UI can strand a corrected entry. `MealEntriesList` derives its query date once, from `todayLocal()` on mount, and never changes it. `LogMealForm` derives *its* `entryDate` the same way, independently. The two happen to agree on a fresh page load, so nobody noticed — until AC-M19 let a user move an entry's date. The moment an entry is re-attributed to any day but today, the list (still pinned to today) can't show it, and there is no in-UI way to browse to that day. The entry is correct in the database — `/daily` scrubbed to that day proves it — but from `/log-meal` it is now unreachable for a second edit or a delete, short of a direct DB write. Because edit/delete is the trust mechanism for the totals and the strip, a correction path that can hide its own results undercuts the very thing Edit & Delete Meal shipped to guarantee.

## 3. Objective

`/log-meal` has one date context. Whatever day the form is set to, the entries list below shows that day's meals — and when a save (new or edit) completes, the page settles on the saved entry's date so the just-touched entry is right there. Moving an entry to an earlier day (AC-M19) no longer costs you the path back to it: the list follows to Jul 8 and the entry is immediately available to edit again or delete. Success = no logged entry is ever reachable in the database but unreachable through the `/log-meal` UI; the "one-way date edit" trap is gone, with no new surface and no new server code.

## 4. Functional Requirements — user-centered acceptance criteria

Written as testable Given/When/Then. "The user" = Albert (single user). New ids continue AC-M25+.

### 4.1 One shared date context

* AC-M25 — Given `/log-meal` renders, When the page mounts, Then the form's date field and the entries-list heading/query resolve to the **same** date (initially `todayLocal()`), sourced from a single owner (`LogMealPanel`), not from two independent `todayLocal()` calls. There is exactly one date input on the page (the existing one in the form).
* AC-M26 — Given the page is on some date D, When the user changes the form's date input to a different day D′, Then the entries list below re-queries and re-renders D′'s logged meals (via the existing `listMealEntriesForDate`), and the list heading reflects D′ — with no page reload.

### 4.2 Save snaps the page to the saved date

* AC-M27 — Given the user edits an entry's date from today to an earlier day D′ and Saves, When the save completes, Then the page's date context becomes D′, the entries list shows D′'s meals including the just-moved entry, and that entry exposes Edit and Delete exactly as any other row (closing the reachability gap; DL-2026-07-21-a2). The re-attribution recompute itself (AC-M19) is unchanged and still correct on both days.
* AC-M28 — Given the user logs a **new** entry for a non-today date D′ (having set the form's date to D′), When the save completes, Then the page stays on D′ and the new entry appears in the list below — the list is not silently thrown back to today. (Note the pre-existing create-mode behavior this slice does **not** change: after a create save the form does not remount and does not reset the meal-type or macro fields — it only mints a fresh idempotency key. AC-M28's assertion is about the page's *date context* only; do not write a test that expects a cleared form.)

### 4.3 Heading honesty

* AC-M29 — Given the entries list is showing date D, When D equals `todayLocal()`, Then the heading reads "Today's meals"; When D is any other day, Then the heading reads "Meals · {D}" with D as the ISO `YYYY-MM-DD` string (an ISO date is an acceptable, deliberate choice for this single-user tool — a friendlier long-form date is an out-of-scope fast-follow, not a build-agent judgment call). The empty state ("Nothing logged yet.") and the phone-usable row layout (AC-M16) are unchanged.

### 4.4 No regression to the correction path or downstream surfaces

* AC-M30 — Given any edit or delete completes on a date-scoped list (D ≠ today), Then all Edit & Delete Meal behaviors hold unchanged: the same row is updated in place / hard-deleted (AC-M18/M21), both affected days' totals, calories-strip bars, and coloring recompute (AC-M19/M22/M23), and unrelated days, Day Strain, badge, guardrails, ledger, and the five-chart count are unregressed (AC-M24). This slice changes only *which day the list is looking at*, never how a mutation is persisted or recomputed.
* AC-M31 — Given the SSR/hydration boundary, When the page first renders on the server (where device-local "today" is not knowable), Then the date resolves to null until the client mounts and sets `todayLocal()` once (preserving the existing hydration-mismatch guard), and neither the form nor the list flashes a wrong day or throws a hydration warning. (Deliberate first-paint change: with the single date owner in `LogMealPanel` gating on `entryDate === null`, the whole panel renders nothing for one mount tick and then appears as a unit — replacing the prior behavior where the form rendered immediately with a blank date field. This is intended and is what makes the no-wrong-day-flash guarantee clean; it is not a regression to "fix.")

## 5. Non-Functional Requirements (FDE)

Aligned to [[JerkAI - Architecture & Data Model]]. All prior NFRs (1–39) remain in force where applicable; the following are additive, numbered continuing from the Edit & Delete Meal PRD.

* NFR-40 — Single source of truth for page date; no duplicated `todayLocal()` ownership. The date lives in exactly one owner (`LogMealPanel` state) and is threaded to form and list as props. The `MealEntriesList` mount effect that independently seeds `todayLocal()` is removed — the list becomes a controlled reader of the panel's date. No third copy of the date is introduced anywhere.
* NFR-41 — Reuse the existing read path, add no server surface. Date changes re-query through the existing `listMealEntriesForDate(entryDate)` "use server" function — no new route handler, no new server action, no client fetch/SWR layer (there is none in this codebase and this slice does not add one). Auth on that function is unchanged (it already returns `[]` for an unauthenticated session).
* NFR-42 — Stale-fetch safety on rapid date changes. When the user changes the date several times quickly (or a save-driven snap races an in-flight fetch), the list must render the entries for the *currently selected* date, never a late-arriving response for a prior date. Preserve/extend the existing `cancelled` guard in the list's fetch effect so out-of-order responses are dropped. Integration- or component-tested for the change-D-then-change-D′-before-first-resolves ordering.
* NFR-43 — No new render-perf or query cost per day. Switching days issues exactly one `listMealEntriesForDate` call for the newly selected day (the same call the list already makes for today); no N+1, no prefetch of adjacent days, no denormalized aggregate. Recompute of totals/strip remains derived downstream at read time (NFR-35, unchanged).
* NFR-44 — Test continuity, with a new interactive tier (DL-2026-07-21-b). Unit-test it there (cheap, and good design regardless). AC ids in every test name. Required fixtures: form-date change re-queries the list to the new day (AC-M26, interactive); a date-edit save snaps the page to the saved date and the moved entry is present with Edit/Delete (AC-M27, interactive); a new non-today entry keeps the page on that date (AC-M28, interactive); heading reads "Today's meals" only when D === today (AC-M29, unit on the extracted formatter + one interactive assertion); null-until-mounted hydration guard holds with no wrong-day flash (AC-M31, interactive); **rapid date-change / save-snap drops the stale in-flight response (NFR-42, interactive — this is the load-bearing test of the slice, exercising the change-D-then-change-D′-before-first-resolves ordering).** No migration fixture (no schema change).

## 9. Implementation notes (build agent — read before touching the three components)

These are not new requirements; they capture the non-obvious mechanics of the lift so a build agent does not regress existing behavior. The three files are `app/ui/log-meal-panel.tsx` (new single date owner), `app/ui/meal-entries-list.tsx` (becomes a controlled reader), `app/ui/log-meal-form.tsx` (create-mode date input becomes controlled by props).

* **IN-1 — The form's mount effect does three jobs; keep two of them.** `LogMealForm`'s mount effect currently seeds `mealType` (from the device clock), `entryDate`, *and* `idempotencyKey`. This slice moves only the create-mode *date* ownership up to the panel. The mount effect must still run in create mode to seed `mealType` and `idempotencyKey` — do not delete or short-circuit it, or `idempotencyKey` stays null forever and the form never becomes submittable. The only change to the effect is that it stops owning the create-mode date (keep a `?? todayLocal()` fallback only for standalone/test rendering with no parent-supplied date; in production the panel always supplies a non-null date because it gates children on `entryDate !== null`).
* **IN-2 — `ready` must read the effective (prop) date in create mode.** The submit-enable guard's `entryDate !== null` check must reference the create-mode date's new source (the prop), not a leftover local `entryDate` that no longer updates. Split the form's single `entryDate` state cleanly: edit mode keeps a local, save-scoped copy (seeded from `editEntry.entryDate`, unchanged); create mode is fully controlled by the `entryDate` prop + `onDateChange` callback, with no local mirror. The date `<input>`'s `value`/`onChange` branch on mode accordingly; the `key={editEntry?.id ?? "new"}` remount trick is untouched.
* **IN-3 — The AC-M27 snap linchpin is confirmed in the shipped code.** `updateMealEntryAction` returns `state.entryDate = validated.value.entryDate` — i.e. the *newly edited* date, not the entry's original date (`app/log-meal/actions.ts`, success branch). So `onMutationSuccess(state.entryDate)` → `setEntryDate(savedDate)` lands the page on the edited day. Do not add any action-state change to make this work; it already holds. The success-effect narrowing (`state.status === "success" && state.entryDate`) is defensive TS only.
* **IN-4 — One fetch per save, not two.** The panel's success handler calls `setEntryDate(savedDate)` and `setRefreshToken(t => t + 1)` together; React batches them into one re-render, so the list's `[entryDate, refreshToken]` fetch effect fires once (NFR-43). When a new entry is logged on the already-viewed day, `savedDate` equals the current date so `setEntryDate` is a no-op, but the `refreshToken` bump still triggers exactly one refetch to surface the new row.
* **IN-5 — `onEditComplete` must not touch the page date.** On edit save the form fires both `onMutationSuccess(savedDate)` (which moves the page date) and `onEditComplete()` (which clears `editEntry` only). Keep `onEditComplete` limited to clearing edit mode — Cancel also calls it, and Cancel must leave the page date exactly where it was (AC-M20 semantics preserved).
