## 0. Scope & Sequencing Note (read first)

This slice exists because JerkAI is now a public portfolio artifact linked from LinkedIn, and the single most valuable thing a technical viewer can do is *see the product*, not read about it. The hard constraint is non-negotiable and predates this PRD: **no real biometric data is ever exposed publicly; the public demo uses synthetic data** (`docs/context.md`, "What JerkAI is"). This PRD's entire job is to satisfy that constraint *structurally* — the demo surface must be incapable of reading real data, not merely configured not to.

**Decision — Log Meal and Targets are deliberately excluded from the demo (DL-2026-07-22-b, "option 1").** The demo renders the read-only dashboard surfaces only. The `/log-meal` page (meal entry + edit/delete) and Settings → Targets (effective-dated target editing) are write surfaces and are NOT rendered on the demo — not even as inert, non-submitting forms. Rationale: including them would trade the slice's strongest safety property ("structurally can't write" — the demo imports no DB module, NFR-51) for a behavioral "handlers are inert" guarantee, plus real refactor and maintenance cost, in exchange for breadth the public GitHub repo already demonstrates better (the Log Meal PRD, server actions, edit/delete slice, and tests all live there in full). The demo still shows the *outcome* of that data — the calories-vs-target strip renders from the synthetic fixture with over/under coloring and gap days. If a taste of the write-side UI is ever wanted on the demo, the sanctioned form is a single static, non-interactive screenshot (no live form, no write surface), not a rendered form.

**Decision — Path A over Path B (DL-2026-07-22-a).** Two paths were weighed: (A) a dedicated unauthenticated surface reading a separate synthetic source, and (B) a "demo mode" toggle inside the existing authenticated app that flips the data source. Path B was rejected: it puts "bypass the gate" logic inside the same app that serves real biometrics, so a single mis-scoped query or half-switched session state becomes a direct real-data leak — it buys convenience by weakening the one invariant that matters most. Path A keeps the auth gate absolute ("session ⇒ real data, always; no session ⇒ demo route, synthetic only") and tells a cleaner data-separation story to the technical audience now inspecting the repo. This PRD builds Path A.

**Structural isolation principle (the crux of the whole slice).** The demo route MUST NOT be able to reach the production database at all — not "is configured to query synthetic rows," but *cannot import or invoke any module that opens a DB connection*. The synthetic data is a committed in-repo fixture module; the demo route reads from it and from nothing else. This is a stronger guarantee than a separate Neon branch (which still routes through live DB code and a connection string that could be mis-pointed): a route that never imports `lib/db.ts` or any query module physically cannot serve a real row, and that property is checkable by a test (AC-PD5 / NFR-51).

What this slice is NOT: a new dashboard feature (the demo renders the *existing* `/daily` and `/weekly` surfaces unchanged); any write path (the demo is strictly read-only — Log Meal, Whoop connect, and any mutation CTA are absent or inert on the demo surface, AC-PD4); any change to the authenticated app's routing, gate, or data path (the real app behaves exactly as today, AC-PD6); or a real-data seed of any kind.

## 2. Problem statement

JerkAI is now linked publicly from Albert's LinkedIn as portfolio and code. A recruiter, hiring manager, or technical screener who clicks through today hits the Auth.js magic-link wall (`proxy.ts` gates the whole app; magic links deliver only to the account owner via Resend, `auth.ts`) and sees nothing of the product. The artifact that best demonstrates "metric tree → spec → production, owned solo" is invisible to exactly the audience it was built to reach. There is no publicly reachable rendering of the dashboard, and there must never be one that touches real biometric data.

## 3. Objective

A stranger with only the URL `demo.jerkai.app` sees the real JerkAI dashboard — the `/weekly` Weekly Ledger and the `/daily` strip stack, rendering the full driver tree (body-fat trend, weight, day strain, calories-vs-target, lean mass, guardrail readouts) — populated with realistic **synthetic** data that shows the product's whole point (a trend that stalls, drivers that co-move, gaps where a day has no entry). No sign-in, no real data, no write actions. The authenticated app is entirely unchanged. Success = a viewer immediately understands what the product does, and the demo surface is *structurally incapable* of serving a real biometric row.

## 4. Functional Requirements — user-centered acceptance criteria

Written as testable Given/When/Then. "The viewer" = any unauthenticated public visitor. New id prefix **AC-PD** (public demo), numbered from 1 to avoid collision with the AC-D (dashboard) and AC-M (meal) series.

* AC-PD1 — Given an unauthenticated visitor, When they navigate to `demo.jerkai.app` (or the canonical demo path it resolves to), Then the Weekly Ledger renders with synthetic data and **no redirect to `/signin`** occurs — the demo surface is excluded from the session gate.
* AC-PD2 — Given the viewer is on the demo, When they follow the Weekly→Daily navigation into the `/daily` drill-down, Then the full strip stack renders over synthetic data on the shared date axis, hover-scrub crosshair included, exactly as the authenticated `/daily` does (same components, different data source).
* AC-PD3 — Given the synthetic dataset, Then it demonstrates the product's core value on sight: at least one stall/recovery in the body-fat trend so the hero stall badge computes a real state (not the cold-start fallback), driver co-movement (a strain/energy change that lines up with the trend), and at least one day with no logged meal so the calories strip shows a gap (not a zero), per the rendering rules in `docs/context.md`.
* AC-PD4 — Given the viewer is anywhere on the demo surface, Then no write path is reachable: the `/log-meal` page and the Settings → Targets page are **not rendered on the demo at all** (deliberately excluded per DL-2026-07-22-b — not present, not even as inert forms), and any mutation CTA (`+ Log meal`, Whoop connect, target edit, edit/delete on an entry) is either absent or inert. No request from the demo can create, edit, or delete a row anywhere. The demo is strictly read-only.
* AC-PD5 — Given the demo route module tree, Then it does not import `lib/db.ts` or any module that opens a database connection (verified by NFR-51's test) — the surface reads only the committed synthetic fixture, so it *cannot* serve a real biometric row even if misconfigured.
* AC-PD6 — Given the authenticated app, When Albert signs in and uses `/weekly`, `/daily`, `/log-meal`, Settings, ingest, and Whoop as before, Then behavior is byte-for-byte unchanged by this slice — the demo is additive and touches none of the gated app's routing, gate, or data path.
* AC-PD7 — Given a search engine or link-preview crawler, Then the demo is clearly labeled a demo on synthetic data (a visible, non-intrusive "Demo — synthetic data, not real biometrics" marker on the surface), so no viewer mistakes it for real personal health data. (Reinforces the public-data guarantee at the presentation layer.)

## 5. Non-Functional Requirements (FDE)

Aligned to [[JerkAI - Architecture & Data Model]]. All prior NFRs remain in force where applicable; the following are additive, continuing the series after NFR-48 (Meal-Type Edit Staleness).

* NFR-49 — **Gate integrity.** Extend the `proxy.ts` matcher to exclude the demo route subtree, using the same exact-match discipline already applied to `privacy$` (so `/demo` and `/demo/*` are reachable but no unintended sibling path is opened). Do not weaken any existing exclusion. Re-derive the matcher regex carefully — a matcher mistake here is a real-data exposure, so treat it with the same care as the `privacy$`/`api/whoop/callback$` exact matches already in the file. Add an adjacent code comment explaining the demo exclusion, matching the file's existing documentation style.
* NFR-50 — **Component reuse, data-layer swap.** The demo MUST render the *same* Weekly Ledger and Daily strip components as the authenticated app — no forked UI — fed by the synthetic fixture instead of the DB view models. If the current data path couples fetching to rendering, factor out a thin seam (a pure "rows/inputs → view model" function the DB path and the demo path both call); the DB-touching fetch stays only on the authenticated path. Minimize the refactor to exactly this seam; do not restructure the dashboard.
* NFR-51 — **Isolation is tested, not asserted.** Add a test that fails if any module reachable from the demo route's entry imports `lib/db.ts` or the Neon pool (static import-graph assertion, or an equivalent guard). This is the machine-checkable form of AC-PD5 and the whole slice's safety case — it must be green in CI and must actually fail if someone later wires the demo to the DB.
* NFR-52 — **Synthetic fixture is committed and deterministic.** The synthetic dataset lives in-repo (e.g. `lib/demo/synthetic-data.ts` or a JSON fixture beside it), is deterministic (no per-request randomness that could make the demo flap), and is shaped to satisfy AC-PD3. It contains only invented values — no real reading, ever, even historically. Document at the top of the fixture that it is synthetic and public.
* NFR-53 — **Domain routing.** `demo.jerkai.app` resolves to the demo surface via Vercel domain config + a Next rewrite (or the equivalent host-based routing the current stack supports), staying on `main`/Production. Preview deployments need no separate demo domain. Confirm the rewrite does not shadow any existing route. Document the mapping in `README.md`'s environments section.
* NFR-54 — **Crawl posture: allow indexing (DL-2026-07-22-c).** Decision locked — the demo is served **index-allowed**, no `noindex`, because it is JerkAI's only public search presence (the real app is fully auth-gated) and the whole point is portfolio discoverability. Two requirements attach: (a) keep AC-PD7's on-surface "Demo — synthetic data" marker; and (b) **the synthetic-data label must also reach the search snippet** — set the demo's page-level `<title>` and meta description (via the App Router `metadata` export, not a raw element) to name it a demo on synthetic data (e.g. title "JerkAI — Live Demo (synthetic data)"), so a searcher landing cold, without the on-page banner in view, still sees the context. `noindex` is a reversible one-line addition later if ever wanted; state the chosen posture in the PR regardless.
* NFR-55 — Test continuity. Reuse the existing Vitest projects (`unit`, `component`, `integration`) wired in `vitest.config.ts` — do not create new ones. Demo-render assertions (a strip/ledger renders from the fixture; no `/signin` redirect logic on the demo path) belong in `component`; the import-graph isolation guard (NFR-51) belongs where it can see the module graph (`unit`/`integration` as convenient). AC ids in every test name.

## 9. Implementation notes (build agent)

* **IN-1 — Find the data seam first (do this before writing routes).** Trace how `/weekly` and `/daily` currently get their data: which `lib/` functions they call, where the DB read happens (`lib/db.ts` / the dashboard `lib/dashboard/*` builders), and where raw rows become the view models the components render. The demo reuses everything from the view-model boundary *up* and substitutes the fixture for everything below it. If fetch and shape are already separated, the demo just calls the shaping functions with fixture rows. If they are coupled, factor out the minimal pure seam per NFR-50 — no more.
* **IN-2 — The matcher edit (security-sensitive, mirror the existing style).** In `proxy.ts`, add the demo path to the negative-lookahead matcher alongside the existing exclusions (`api/ingest`, `api/auth`, `api/whoop/callback$`, `api/whoop/sync$`, `signin`, `privacy$`, static assets). Use the same exact-match discipline the file already applies (`privacy$` excludes `/privacy` only, not `/privacy-*`) so the demo exclusion opens exactly the demo subtree and nothing else. Add a comment in the file's existing explanatory style documenting why the demo path is public. Pages already re-check the session themselves as defense in depth (`app/page.tsx`, `app/status/page.tsx`) — the demo pages deliberately do the opposite: they never call `auth()` and never import a DB module, so there is nothing to bypass.
* **IN-3 — The isolation guard (NFR-51) is the safety case.** Write it so it fails loudly if the demo ever imports a DB path: a static import-graph assertion from the demo route entry (walk the import tree, assert `lib/db.ts` and the Neon pool are absent), or the equivalent. This test is the machine-checkable proof of the whole slice's promise — treat a failure of it as a release blocker, not a flake.
* **IN-4 — Synthetic fixture shape.** Build the fixture to satisfy AC-PD3 deliberately: enough completed ISO weeks for the Weekly Ledger to show real deltas (aim past the 2-week cold-start threshold so the hero badge computes from completed rows, not the daily-streak fallback, per `docs/context.md` DL-2026-07-19-b), a body-fat trend that visibly stalls then resumes, a strain or calories series that co-moves with the stall, and at least one day with no meal entry so the calories strip renders a gap rather than a zero. Deterministic values only; header-comment the file as synthetic/public. No real reading may appear in it, now or ever.
* **IN-5 — Read-only enforcement (AC-PD4).** The demo renders the display components but not the mutation affordances: omit or inert the `+ Log meal` CTA, Whoop connect, target editing, and per-entry edit/delete on the demo surface. The cleanest form is that the demo route simply never mounts those components; if a shared component carries a CTA, pass it into a read-only mode rather than forking it. No demo request should reach any write handler — and since the demo imports no DB module (IN-3), it structurally can't.
